# 데이터베이스 스키마 명세서 — 발열 Yes/No 100점 채점 반영판

## 1. 개요 및 Supabase 보안 모델

본 문서는 AI 표준화환자 CPX 플랫폼의 데이터베이스 구조를 정의한다. Supabase(PostgreSQL)를 기반으로 하며, 일반 CRUD는 프론트엔드가 Supabase SDK로 직접 수행하고, AI 환자 대화 및 자동 채점은 FastAPI가 처리한다.

## 공통 채점 기준 — 발열 채점표 Yes/No 100점

- 기준 문서: `CPX_발열_채점표_YesNo_100점.md`
- 총 항목 수: 40개
- 판정값: `Yes` 또는 `No`
- 가중치: 없음
- 부분점수: 없음
- 최종 점수: `Yes 개수 / 40 × 100`
- 항목당 환산 점수: 2.5점
- 기존 `잘함/보통/미흡`은 사용하지 않음


### 보안 모델
- **RLS(Row Level Security)**: 학생은 본인의 세션과 결과만 조회한다.
- **Auth 연동**: Supabase `auth.users`와 `public.Users`를 Trigger로 동기화한다.
- **평가 결과 보호**: `Feedback_Results`는 학생 본인, 담당 교수, 관리자만 조회할 수 있다.

## 2. 테이블 명세

### 2.1 Organizations — 기관 및 학교
B2B 고객사 관리를 위한 테이블이다.

| 필드 | 타입 | 설명 |
|---|---|---|
| `organization_id` | UUID PK | 기관 고유 ID |
| `name` | Text | 기관명 |
| `domain` | Text | 기관 이메일 도메인 |
| `created_at` | Timestamp | 등록일 |

### 2.2 Subscriptions — 구독 및 결제
기관 또는 개인 사용자의 구독 상태를 관리한다.

| 필드 | 타입 | 설명 |
|---|---|---|
| `subscription_id` | UUID PK | 구독 ID |
| `organization_id` | UUID FK, nullable | 기관 ID |
| `user_id` | UUID FK, nullable | 개인 사용자 ID |
| `plan_type` | Enum | `BASIC`, `PREMIUM`, `ENTERPRISE` |
| `status` | Enum | `ACTIVE`, `EXPIRED`, `CANCELED` |
| `start_date` | Date | 시작일 |
| `end_date` | Date | 종료일 |

### 2.3 Users — 사용자
의대생, 교수, 관리자의 계정 정보를 저장한다.

| 필드 | 타입 | 설명 |
|---|---|---|
| `user_id` | UUID PK | 사용자 고유 ID. Supabase Auth UID와 연결 |
| `organization_id` | UUID FK, nullable | 소속 기관 ID |
| `email` | Text unique | 로그인 이메일 |
| `name` | Text | 사용자 이름 |
| `grade` | Integer, nullable | 학년 |
| `role` | Enum | `STUDENT`, `PROFESSOR`, `ADMIN` |
| `created_at` | Timestamp | 가입일 |

> Supabase Auth를 쓰므로 `password_hash`는 `public.Users`에 저장하지 않는다.

### 2.4 Scenarios — CPX 시나리오
진료 연습의 환자 시나리오 데이터다.

| 필드 | 타입 | 설명 |
|---|---|---|
| `scenario_id` | UUID PK | 시나리오 ID |
| `title` | Text | 시나리오 제목 |
| `chief_complaint` | Text | 주소증 |
| `department` | Text | 진료 과목 |
| `symptom_tag` | Text | 예: `발열` |
| `difficulty` | Enum | `EASY`, `MEDIUM`, `HARD` |
| `patient_info` | JSONB | 이름, 나이, 성별, 활력징후, 과거력 등 |
| `positive_findings` | JSONB | 케이스별 양성 소견 |
| `negative_findings` | JSONB | 케이스별 음성 소견 |
| `exam_findings` | JSONB | 진찰 요청 시 반환할 소견 |
| `education_points` | JSONB | 환자교육 핵심 내용 |
| `is_active` | Boolean | 활성화 여부 |
| `created_by` | UUID FK | 생성자 ID |
| `created_at` | Timestamp | 생성일 |

### 2.5 Rubrics — 채점 루브릭
발열 채점표 40개 Yes/No 항목을 저장한다.

| 필드 | 타입 | 설명 |
|---|---|---|
| `rubric_id` | UUID PK | 루브릭 ID |
| `scenario_id` | UUID FK, nullable | 특정 시나리오 전용이면 연결, 공통 발열 루브릭이면 nullable 가능 |
| `symptom_tag` | Text | 예: `발열` |
| `version` | Integer | 루브릭 버전 |
| `scoring_mode` | Text | `fever_checklist_yes_no_100_no_weight` |
| `total_items` | Integer | 기본값 40 |
| `criteria` | JSONB | 40개 평가 항목 |
| `is_active` | Boolean | 현재 사용 여부 |
| `updated_by` | UUID FK | 마지막 수정자 |
| `created_at` | Timestamp | 생성일 |

#### `Rubrics.criteria` JSONB 구조

```json
{
  "scoring_mode": "fever_checklist_yes_no_100_no_weight",
  "total_items": 40,
  "score_formula": "yes_count / 40 * 100",
  "uses_weight": false,
  "uses_partial_score": false,
  "result_type": "YES_NO",
  "items": [
    {
      "id": 1,
      "domain": "병력 청취-주제 관련",
      "criterion": "발열이 시작된 시기와 지속 기간을 질문하였다.",
      "result_type": "YES_NO",
      "evidence_required": true
    }
  ]
}
```

### 2.6 Sessions — 연습 세션
사용자가 연습한 한 회차의 정보를 기록한다.

| 필드 | 타입 | 설명 |
|---|---|---|
| `session_id` | UUID PK | 세션 ID |
| `user_id` | UUID FK | 사용자 ID |
| `scenario_id` | UUID FK | 시나리오 ID |
| `rubric_id` | UUID FK | 평가에 사용할 루브릭 ID |
| `mode` | Enum | `PRACTICE`, `ACTIVE`, `EXAM` |
| `start_time` | Timestamp | 시작 시간 |
| `end_time` | Timestamp, nullable | 종료 시간 |
| `status` | Enum | `IN_PROGRESS`, `COMPLETED`, `EVALUATING`, `EVALUATED`, `ERROR` |

### 2.7 Transcripts — 대화 전사 및 오디오 기록
세션 중 발생한 학생/AI 발화를 저장한다.

| 필드 | 타입 | 설명 |
|---|---|---|
| `transcript_id` | UUID PK | 전사 ID |
| `session_id` | UUID FK | 세션 ID |
| `speaker` | Enum | `USER`, `AI`, `SYSTEM` |
| `content` | Text | 발화 텍스트 |
| `audio_url` | Text, nullable | 음성 파일 경로 |
| `event_type` | Enum | `STT_RESULT`, `AI_REPLY`, `TUTOR_GUIDE`, `EXAM_ACTION`, `EDUCATION` |
| `timestamp` | Timestamp | 발화 발생 시간 |

> 채점에서 `Yes` 근거로 사용할 1차 데이터는 `speaker=USER` 발화다.

### 2.8 Feedback_Results — 피드백 및 평가 결과
세션 종료 후 Yes/No 자동 채점 결과를 저장한다.

| 필드 | 타입 | 설명 |
|---|---|---|
| `result_id` | UUID PK | 결과 ID |
| `session_id` | UUID FK unique | 세션 ID |
| `rubric_id` | UUID FK | 사용한 루브릭 버전 |
| `scoring_mode` | Text | `fever_checklist_yes_no_100_no_weight` |
| `total_items` | Integer | 기본값 40 |
| `yes_count` | Integer | Yes 항목 수 |
| `no_count` | Integer | No 항목 수 |
| `total_score` | Numeric(5,2) | `yes_count / 40 × 100` |
| `score_items` | JSONB | 40개 항목별 Yes/No, 근거, 피드백 |
| `missed_items` | JSONB | No 항목만 모은 배열 |
| `domain_summary` | JSONB | 영역별 Yes/No 개수. 참고용이며 가중치 없음 |
| `strengths` | JSONB | 잘한 항목 요약 |
| `weaknesses` | JSONB | 개선 필요 항목 요약 |
| `clinical_reasoning_flow` | JSONB | 질문 흐름 시각화 데이터. 점수 가중치 없음 |
| `explainable_feedback` | JSONB | 왜 필요한지 설명하는 피드백 |
| `created_at` | Timestamp | 생성 시간 |

## 5. 구현 시 주의사항

- `score_history_taking`, `score_communication`, `score_education`처럼 영역별 점수 필드를 둘 수는 있지만, 최종 점수는 반드시 `yes_count / 40 × 100`만 사용한다.
- `잘함`, `보통`, `미흡`, `weight`, `partial_score` 필드는 사용하지 않는다.
- 평가 엔진은 항목별 근거 발화를 반드시 저장해야 한다.
- 루브릭 버전이 바뀌어도 과거 결과는 당시 `rubric_id`를 기준으로 재현 가능해야 한다.
