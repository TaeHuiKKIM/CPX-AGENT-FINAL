# 백엔드 개발 태스크 칸반 — 발열 Yes/No 100점 채점 반영판

본 문서는 **Supabase(BaaS) + FastAPI(Microservice)** 하이브리드 아키텍처를 기준으로 백엔드 개발 태스크를 정리한다. 기존 문서의 3개 Epic 구조는 유지하되, 평가 엔진은 전부 **발열 채점표 예시 기반 Yes/No 100점 방식**으로 수정한다.

## 공통 채점 기준 — 발열 채점표 Yes/No 100점

- 기준 문서: `CPX_발열_채점표_YesNo_100점.md`
- 총 항목 수: 40개
- 판정값: `Yes` 또는 `No`
- 가중치: 없음
- 부분점수: 없음
- 최종 점수: `Yes 개수 / 40 × 100`
- 항목당 환산 점수: 2.5점
- 기존 `잘함/보통/미흡`은 사용하지 않음

---

## Epic 1. Supabase 인프라 구축 — DB, Auth, Storage
> **Goal**: 프론트엔드가 안전하게 데이터에 접근하고, 평가 엔진이 40개 Yes/No 결과를 저장할 수 있는 DB 구조를 만든다.

### Task 1-1. Supabase 프로젝트 생성 및 SQL 배포
- `Organizations`, `Users`, `Scenarios`, `Sessions`, `Transcripts`, `Rubrics`, `Feedback_Results` 테이블을 생성한다.
- `Rubrics.criteria`에는 발열 채점표 40개 항목을 JSONB로 저장한다.
- `Rubrics.scoring_mode` 기본값은 `fever_checklist_yes_no_100_no_weight`로 둔다.

### Task 1-2. RLS 정책 설정
- 학생은 본인의 `Sessions`, `Transcripts`, `Feedback_Results`만 조회 가능하다.
- 교수/관리자는 소속 기관의 시나리오, 루브릭, 결과 데이터를 조회 가능하다.
- 루브릭 수정은 `PROFESSOR` 또는 `ADMIN`만 가능하다.

### Task 1-3. Supabase Auth 연동
- 이메일 로그인/회원가입을 활성화한다.
- `auth.users`와 `public.Users`를 Trigger로 동기화한다.
- 사용자의 `role`에 따라 학생/교수/관리자 권한을 분기한다.

### Task 1-4. Storage 버킷 생성
- 진료 음성 파일 저장용 `audio_records` 버킷을 생성한다.
- 파일 경로는 `Transcripts.audio_url`에 저장한다.

### Task 1-5. 발열 Yes/No 루브릭 Seed 데이터 삽입
- 40개 항목을 `Rubrics.criteria.items[]`에 삽입한다.
- 각 항목은 `id`, `domain`, `criterion`, `result_type`, `evidence_required`를 가진다.
- 모든 항목의 `result_type`은 `YES_NO`로 통일한다.
- `weight`, `partial_score`, `level` 필드는 사용하지 않는다.

## Epic 2. FastAPI 실시간 AI 통신 엔진 구축 — WebSocket
> **Goal**: 실시간 대화 중 AI 환자가 정보를 먼저 누설하지 않도록 제어하고, 이후 채점 가능한 Transcript를 정확히 남긴다.

### Task 2-1. FastAPI 프로젝트 초기화
- `fastapi`, `uvicorn`, `supabase-py`, STT/TTS 클라이언트를 세팅한다.
- 환경변수로 Supabase URL, Service Role Key, LLM API Key를 관리한다.

### Task 2-2. LLM 프롬프트 빌더 구현
- `Scenarios.patient_info`와 `Rubrics.criteria`를 읽어 시스템 프롬프트를 생성한다.
- 환자 프롬프트에는 “묻지 않은 정보 선제 제공 금지”를 최상단 규칙으로 넣는다.
- 튜터 힌트는 Practice Mode에서만 반환하고, Active/Exam Mode에서는 반환하지 않는다.

### Task 2-3. WebSocket 라우터 구현
- 엔드포인트: `WebSocket /api/v1/sessions/{session_id}/stream`
- Client는 오디오 청크를 전송한다.
- Server는 STT 결과, AI 환자 응답, TTS 오디오를 반환한다.
- 모든 사용자 발화와 AI 응답은 `Transcripts`에 저장한다.

### Task 2-4. STT/LLM/TTS 파이프라인 구현
- 오디오 → STT → LLM 환자 응답 → TTS 순서로 처리한다.
- STT 결과가 채점 근거가 되므로 원문을 최대한 보존한다.
- AI 환자 응답은 채점 항목의 Yes 근거로 사용하지 않는다. 학생의 질문/진찰/교육 발화만 Yes 근거로 사용한다.

### Task 2-5. Transcript 저장 품질 관리
- `speaker=USER` 발화는 학생 수행 판단의 1차 근거다.
- `speaker=AI` 발화는 환자 반응 및 맥락 확인용으로만 사용한다.
- 각 발화에는 `timestamp`, `content`, `audio_url`, `event_type`을 저장한다.

## Epic 3. 평가 엔진 및 Explainable Feedback — FastAPI
> **Goal**: 세션 종료 시 Transcript와 40개 Yes/No 루브릭을 대조해 100점 환산 점수와 누락 이유 피드백을 생성한다.

### Task 3-1. Yes/No 루브릭 로더 구현
- `Rubrics.criteria.items[]`에서 40개 항목을 불러온다.
- 항목 수가 40개가 아니면 평가를 중단하고 관리자 오류를 반환한다.
- `scoring_mode`가 `fever_checklist_yes_no_100_no_weight`인지 검증한다.

### Task 3-2. Transcript 기반 항목별 판정 알고리즘 구현
- 학생 발화(`speaker=USER`)만 기준으로 각 항목을 판정한다.
- 항목별 근거 문장이 있으면 `Yes`, 없으면 `No`로 저장한다.
- 의미상 같은 질문은 표현이 달라도 `Yes`로 인정한다.
- 모호한 발화, 환자가 먼저 말한 정보, 부분 수행은 `No`로 처리한다.

### Task 3-3. 100점 환산 점수 계산
```python
yes_count = sum(1 for item in items if item["result"] == "Yes")
total_items = 40
score_total = round((yes_count / total_items) * 100, 1)
```
- 항목별 가중치는 계산하지 않는다.
- 영역별 점수는 참고 표시용이며 최종 점수 산식에는 반영하지 않는다.

### Task 3-4. Explainable Feedback 생성
- `No` 항목을 중심으로 피드백을 생성한다.
- 각 피드백은 `누락한 항목`, `왜 필요한지`, `다음 실습에서 쓸 수 있는 예시 질문`을 포함한다.
- 예: 여행력 미확인 → “말라리아, 장티푸스 등 감염성 발열 감별을 위해 필요합니다.”

### Task 3-5. Clinical Reasoning 시각화 데이터 생성
- 학생 질문 흐름을 `timeline` 또는 `flow_nodes`로 저장한다.
- 단, 임상추론 시각화는 참고용 UI 데이터이며 점수 산식에는 가중치로 반영하지 않는다.

### Task 3-6. 결과 저장
- `Feedback_Results`에 `yes_count`, `no_count`, `total_items`, `total_score`, `score_items`, `missed_items`, `domain_summary`, `explainable_feedback`을 저장한다.
- 기존 `score_history_taking`, `score_communication`, `score_education`은 사용하더라도 영역별 참고 점수로만 취급한다.

## Epic 4. 프론트엔드 연동 체크리스트 — 평가 결과 UI
> **Goal**: 사용자가 결과 화면에서 100점 점수와 40개 Yes/No 항목을 한눈에 확인할 수 있게 한다.

### Task 4-1. 결과 요약 카드
- 총점: `score_total / 100`
- Yes/No 개수: `yes_count / no_count`
- 채점 방식: `가중치 없음 · 40항목 Yes/No`

### Task 4-2. 40개 항목 테이블
- 컬럼: `No`, `영역`, `평가 항목`, `Yes/No`, `근거 발화`, `피드백`
- `No` 항목은 강조 표시한다.

### Task 4-3. 누락 항목 코칭 카드
- 병력청취, 신체진찰, 환자교육, PPI별로 누락 항목을 묶어 보여준다.
- “왜 필요한지”를 함께 표시한다.
