# 해커톤 기술 질의응답 Q&A

발표 후 심사위원이 기술적인 부분을 물어봤을 때 바로 답할 수 있도록 정리한 문서입니다. 답변은 너무 어렵게 말하지 않고, “왜 그렇게 만들었는지”와 “현재 어디까지 구현됐는지”를 중심으로 말하면 됩니다.

## 1. 전체 구조

### Q1. 전체 아키텍처가 어떻게 구성되어 있나요?

**답변**

프론트엔드는 React/Vite로 만들었고, 백엔드는 FastAPI로 구성했습니다. 사용자는 브라우저에서 문진과 신체진찰을 진행하고, 백엔드는 WebSocket으로 AI 환자 응답을 처리합니다. 로그인과 데이터베이스는 Supabase를 사용했고, 배포는 Cloudtype에서 Dockerfile 하나로 프론트와 백엔드를 같이 서빙하는 방식입니다.

**짧게 말하면**

React 프론트, FastAPI 백엔드, Gemini AI, Supabase DB/Auth를 Cloudtype에 한 번에 배포한 구조입니다.

### Q2. 왜 프론트와 백엔드를 하나의 Dockerfile로 배포했나요?

**답변**

해커톤 데모에서는 배포 안정성과 설정 단순성이 중요했습니다. 프론트를 따로, 백엔드를 따로 배포하면 API 주소와 WebSocket 주소 설정이 복잡해질 수 있어서, FastAPI가 정적 프론트 파일과 API/WebSocket을 함께 서빙하도록 구성했습니다.

**짧게 말하면**

팀원이 접속해서 바로 테스트할 수 있게 배포 설정을 단순하게 만든 선택입니다.

### Q3. WebSocket은 어디에 쓰이나요?

**답변**

문진 중 학생이 질문을 보내면 AI 환자가 실시간으로 답해야 해서 WebSocket을 사용했습니다. 일반 REST API보다 대화형 세션에 자연스럽고, 이후 신체진찰 결과도 같은 세션 맥락에 전달할 수 있습니다.

**짧게 말하면**

AI 환자와 실시간으로 대화하기 위해 WebSocket을 사용했습니다.

## 2. AI 환자와 프롬프트

### Q4. AI 환자가 정답을 먼저 말해버리면 채점이 이상해지지 않나요?

**답변**

그래서 AI 환자 프롬프트에 “학생이 묻지 않은 정보는 먼저 말하지 않는다”는 규칙을 넣었습니다. 예를 들어 과거력, 가족력, 여행력, 동반 증상은 학생이 질문해야만 답하도록 설계했습니다. 채점도 환자가 먼저 말한 정보가 아니라 학생이 직접 질문하거나 수행한 기록을 기준으로 판단합니다.

**짧게 말하면**

AI 환자는 먼저 정보를 누설하지 않게 하고, 채점은 학생이 직접 한 질문과 진찰만 기준으로 합니다.

### Q5. Gemini는 어떤 역할을 하나요?

**답변**

Gemini는 두 가지 역할을 합니다. 첫째, AI 표준화환자 응답을 생성합니다. 둘째, 세션 종료 후 학생의 대화 기록과 신체진찰 로그를 보고 발열 체크리스트 중 어떤 항목이 Yes인지 판단합니다.

**짧게 말하면**

Gemini는 환자 응답 생성과 최종 채점 보조에 사용됩니다.

### Q6. Structured Output을 왜 사용했나요?

**답변**

채점 결과를 자유로운 문장으로 받으면 프론트에서 안정적으로 처리하기 어렵습니다. 그래서 Gemini가 정해진 JSON 구조로 Yes 항목 번호와 근거를 반환하도록 했고, 서버가 그 결과를 40개 전체 체크리스트로 복원합니다.

**짧게 말하면**

AI 결과를 JSON으로 고정해서 채점 결과가 흔들리지 않게 하기 위해 사용했습니다.

## 3. 채점 알고리즘

### Q7. 자동 채점은 어떤 기준으로 하나요?

**답변**

발열 CPX 채점표의 40개 항목을 기준으로 합니다. 각 항목은 Yes 또는 No로만 판단하고, 부분점수나 가중치는 없습니다. 최종 점수는 `Yes 개수 / 40 * 100`으로 계산합니다.

**짧게 말하면**

40개 항목 Yes/No 방식이고, Yes 개수로 100점 환산합니다.

### Q8. 왜 부분점수나 가중치를 쓰지 않았나요?

**답변**

해커톤 MVP에서는 채점 기준이 명확하고 설명 가능해야 했습니다. 부분점수나 가중치를 넣으면 채점 이유가 복잡해지고 사람마다 해석이 달라질 수 있습니다. 그래서 실제 체크리스트처럼 했는지 안 했는지를 Yes/No로 단순하게 판단했습니다.

**짧게 말하면**

채점 기준을 명확하고 설명 가능하게 만들기 위해 Yes/No만 사용했습니다.

### Q9. LLM이 40개 항목을 전부 생성하면 느리지 않나요?

**답변**

맞습니다. 그래서 LLM에는 40개 전체를 쓰게 하지 않고, Yes라고 판단한 항목 번호와 짧은 근거만 반환하게 했습니다. 나머지 No 항목은 서버가 체크리스트 기준으로 자동 복원합니다. 이렇게 해서 응답 길이와 채점 시간을 줄였습니다.

**짧게 말하면**

AI는 Yes 항목만 반환하고, 서버가 전체 40개 결과를 복원해서 속도를 줄였습니다.

### Q10. 환자가 말한 내용도 채점 근거가 되나요?

**답변**

원칙적으로 학생의 질문, 설명, 신체진찰 수행 기록이 채점 근거입니다. 환자가 먼저 말한 내용은 학생이 능동적으로 확인한 것이 아니므로 Yes 근거로 보지 않습니다.

**짧게 말하면**

학생이 직접 묻거나 수행한 것만 채점 근거로 봅니다.

## 4. 신체진찰 모듈

### Q11. 신체진찰은 그냥 텍스트로 입력하면 되지 않나요?

**답변**

CPX에서는 무엇을 물었는지뿐 아니라 어떤 순서로, 어떤 도구로, 어떤 체위에서 진찰했는지도 중요합니다. 그래서 텍스트 입력이 아니라 체위, 손소독, 도구, 진찰 항목을 직접 선택하는 플로우형 모듈로 만들었습니다.

**짧게 말하면**

CPX 절차를 더 실제처럼 반영하기 위해 신체진찰을 버튼 기반 플로우로 만들었습니다.

### Q12. 체위 변경이나 청진 시간이 점수에 어떻게 반영되나요?

**답변**

체위를 바꾸면 2초 로딩 UI가 나오고 실제 세션 타이머에서 2초가 차감됩니다. 일반 진찰은 5초, 환자 협조가 필요한 진찰은 10초가 즉시 차감됩니다. 아무 진찰이나 많이 하면 항목은 얻을 수 있어도 시간이 줄어드는 구조입니다.

**짧게 말하면**

진찰을 하면 바로 시간이 줄어들어서, 실제 시험처럼 시간 관리가 필요합니다.

### Q13. 정수아 케이스에서 CVAT 양성은 어떻게 처리되나요?

**답변**

정수아 케이스는 급성 신우신염 케이스라 CVAT를 수행하면 우측 CVA 압통 양성 소견이 기록되도록 했습니다. 이 소견은 대화 기록의 시스템 메시지와 신체진찰 로그에 들어가고, 최종 채점에서도 신체진찰 근거로 사용됩니다.

**짧게 말하면**

정수아에서 CVAT를 하면 우측 CVA 압통 양성이 기록되고 채점에 반영됩니다.

## 5. 데이터베이스와 보안

### Q14. Supabase는 어떤 용도로 사용하나요?

**답변**

Supabase는 로그인/회원가입, 사용자 세션 저장, 대화 기록, 채점 결과 저장을 위해 사용합니다. 프론트에서는 Supabase Auth를 사용하고, 백엔드는 service role key로 필요한 서버 작업을 처리합니다.

**짧게 말하면**

Supabase는 인증과 데이터 저장을 담당합니다.

### Q15. Supabase anon key가 프론트에 있어도 안전한가요?

**답변**

Supabase anon key는 원래 브라우저에서 쓰도록 설계된 공개 키입니다. 대신 Row Level Security, 즉 RLS 정책으로 사용자가 자기 데이터만 접근하도록 막아야 합니다. service role key는 RLS를 우회할 수 있기 때문에 절대 프론트에 노출하지 않고 서버 환경변수에만 둡니다.

**짧게 말하면**

anon key는 공개 가능하지만 RLS로 보호하고, service role key는 서버에만 둡니다.

### Q16. `.env`나 API 키가 GitHub에 올라가면 어떻게 하나요?

**답변**

즉시 해당 키를 폐기하고 새 키를 발급해야 합니다. 이 프로젝트에서는 `.gitignore`로 `.env`가 올라가지 않게 했고, Cloudtype 환경변수에 키를 넣는 방식으로 관리합니다.

**짧게 말하면**

노출되면 바로 키를 재발급하고, 비밀 키는 Git에 올리지 않습니다.

## 6. 배포와 운영

### Q17. Cloudtype에서는 어떻게 배포했나요?

**답변**

GitHub 레포지토리를 Cloudtype에 연결하고, 루트 Dockerfile을 기준으로 빌드했습니다. Dockerfile은 프론트를 빌드한 뒤 FastAPI 서버가 `dist` 정적 파일과 `/api/v1` API, WebSocket을 함께 서빙하도록 구성했습니다. 포트는 8000을 사용합니다.

**짧게 말하면**

Cloudtype에서 Dockerfile 배포를 사용했고, FastAPI가 프론트와 백엔드를 같이 서빙합니다.

### Q18. 환경변수는 어디에 넣나요?

**답변**

Cloudtype 서비스의 Environment Variables에 넣습니다. 프론트에 필요한 값은 `VITE_SUPABASE_URL`, `VITE_SUPABASE_ANON_KEY`이고, 서버 전용 값은 `SUPABASE_KEY`, `GEMINI_API_KEY`입니다.

**짧게 말하면**

Cloudtype 환경변수에 넣고, 서버 비밀 키에는 `VITE_`를 붙이지 않습니다.

### Q19. 배포 후 가장 먼저 확인할 것은 무엇인가요?

**답변**

먼저 `/health`가 정상 응답하는지 봅니다. 그 다음 `/env.js`에서 Supabase URL이 placeholder가 아니라 실제 값인지 확인합니다. 이후 로그인, WebSocket 대화, 종료 후 성과 리포트 이동을 확인합니다.

**짧게 말하면**

`/health`, `/env.js`, 로그인, 대화, 채점 순서로 확인합니다.

## 7. 오류 대응

### Q20. 로그인할 때 `Failed to fetch`가 뜨면 원인이 뭔가요?

**답변**

대부분 프론트가 잘못된 Supabase URL을 보고 있을 때 발생합니다. `/env.js`에 `your-project-url.supabase.co` 같은 placeholder가 남아 있는지 확인해야 합니다. 실제 Supabase URL과 anon key를 Cloudtype 환경변수에 넣고 다시 배포하면 됩니다.

**짧게 말하면**

Supabase URL 환경변수가 잘못 들어간 경우가 많습니다.

### Q21. 종료 후 채점으로 안 넘어가면 어떻게 처리하나요?

**답변**

현재는 채점 API가 실패해도 연습 기록이 사라지지 않게 했습니다. 성과 리포트에 `채점 실패` 복구 리포트가 생성되고, 거기에 오류 원문과 대화 기록, 신체진찰 기록이 남습니다. 그 오류를 보고 Gemini 키, 서버 로그, 채점 API 상태를 확인하면 됩니다.

**짧게 말하면**

실패해도 기록은 보존되고, 성과 리포트에서 오류 원인을 확인할 수 있습니다.

### Q22. `DB Session insert failed` 경고는 무슨 뜻인가요?

**답변**

로그인은 되었지만 Supabase `sessions` 테이블에 세션 저장이 실패했다는 뜻입니다. 주로 스키마가 잘못 적용됐거나 RLS 정책이 부족할 때 발생합니다. 이 앱은 `supabase/schema.sql`을 기준으로 하고, `backend/schema.sql`은 예전 참고용이라 적용하면 안 됩니다.

**짧게 말하면**

Supabase 스키마나 RLS 문제라서 `supabase/schema.sql`을 다시 확인해야 합니다.

### Q23. 503 에러가 뜨면 어떻게 보나요?

**답변**

Cloudtype 서버가 정상 실행 중인지 확인해야 합니다. 포트가 8000인지, Dockerfile Path가 `Dockerfile`인지, `/health`가 응답하는지 봅니다. 서버 로그에서 `GEMINI_API_KEY`, `SUPABASE_URL`, `SUPABASE_KEY` 누락도 확인합니다.

**짧게 말하면**

Cloudtype 실행 로그, 포트 8000, `/health`, 환경변수를 확인합니다.

## 8. 확장성과 한계

### Q24. 발열 말고 다른 케이스로 확장할 수 있나요?

**답변**

가능합니다. 현재는 발열 40개 체크리스트를 기준으로 만들었지만, 기침, 흉통, 복통 같은 다른 주제도 먼저 체크리스트를 구조화하면 같은 방식으로 확장할 수 있습니다. 환자 데이터와 진찰 소견만 바꾸고, 채점 엔진은 Yes/No 구조를 재사용할 수 있습니다.

**짧게 말하면**

체크리스트만 새로 만들면 다른 CPX 주제로 확장할 수 있습니다.

### Q25. 현재 한계는 무엇인가요?

**답변**

현재 MVP는 발열 케이스와 브라우저 기반 음성 기능에 초점을 맞췄습니다. 실제 병원 수준의 음성 품질, 대규모 사용자 동시 접속, 교수자용 통계 대시보드는 추가 개발이 필요합니다. 다만 핵심 흐름인 AI 환자 문진, 신체진찰, 자동 채점, 성과 리포트는 동작하도록 구현했습니다.

**짧게 말하면**

MVP라서 확장 기능은 남아 있지만, 핵심 실습 흐름은 구현되어 있습니다.

### Q26. 의료적으로 안전한가요?

**답변**

이 서비스는 실제 진단용이 아니라 의대생 교육용 CPX 시뮬레이션입니다. 결과도 환자 진단이 아니라 학생이 어떤 문진과 진찰을 수행했는지에 대한 학습 피드백입니다. 따라서 실제 환자 진료 판단에는 사용하지 않는다는 전제를 둡니다.

**짧게 말하면**

진단 도구가 아니라 교육용 시뮬레이션입니다.

- “실제 CPX처럼 시간 손실과 신체진찰 절차를 반영한 것이 차별점입니다.”

## 10. 모르면 이렇게 답하기

### Q. 지금 정확한 수치나 내부 구현을 바로 답하기 어려운 질문을 받으면?
